# AGENTS.md

## 项目使命

Star-in-my-heart 是一个 AI 驱动的 GitHub 仓库"内化导师"。目标不是收集 star，而是通过让用户主动参与，让用户真正理解它们。

**核心理念：AI 不应该提供答案，而应该设计通往真理的路径。**

你作为AI，除了写一些基础的文档帮助用户快速上手之外，还需要引导用户做进一步思考。

## 项目结构

```
├── docs/                    # 文档内容
│   ├── index.md            # 仓库列表表格
```

## 文档风格指南

### 语言与语气

- **主要语言**：简体中文
- **英文**：仅用于技术术语、代码和原始仓库名称
- **语气**：技术性、探索性、个人学习笔记风格
- **目的**：记录**用户的**学习旅程，而非技术规格

### 内容理念（关键）

本项目遵循 **苏格拉底式内化** 原则：

1. **不喂食答案**：不要提供冗长的解释或总结。让读者通过代码自行发现。
2. **输出优先**：留下"思维的面包屑" - 代码注释、实验笔记、重构草稿、**用户的尝试理解**。
3. **破坏性学习**：记录什么会破坏代码以及原因。对代码进行"微型手术"比阅读更有教益。

### 应包含的内容

- **破冰问题**：关于这个仓库，应该问什么问题？
- **考古学家的笔记**：血缘关系、生态系统定位、20/80 规则观察
- **沙盒实验**：你尝试过的代码、哪里出错了、学到了什么
- **费曼检查点**：你能简单解释吗？你在哪里遇到了困难？

### 应避免的内容

- 通用功能列表（从 README 复制的）
- 没有个人见解的长篇技术总结
- 没有注释的代码堆砌
- "完美"文档 - 留下展示学习过程的粗糙边缘

### 流程图

你生成的流程图最好使用mermaid格式，方便渲染。

## DeepWiki 集成

本仓库使用 DeepWiki MCP 获取 GitHub 仓库上下文。代理可以使用 `deepwiki_ask_question` 工具直接查询已加星的仓库。

使用示例：在编写之前询问 Go 仓库，以加深理解。

## 最终准则

1. **先做学习者，再记录**
2. **留下思考过程的痕迹**
3. **问难题，记录挣扎**
4. **展示什么会破坏以及为什么**
5. **当你能简单解释时，再写下来**

> 代码是死的，直到你在脑海中解构、破坏并重建它。
